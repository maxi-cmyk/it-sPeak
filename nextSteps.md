# Plan

Build the PRD's project-folder and progress-tracking system on top of the working multimodal analysis pipeline. The implementation should first make projects, sessions, videos, score snapshots, and coaching cards durable; then add five-session replacement rules, baseline-relative progress, comparable-session safeguards, replay, stagnation alerts, and the coaching playbook without changing the existing CV or audio algorithms.

## Scope
- In: authenticated project ownership, Supabase persistence, Cloudflare R2 video retention, named project folders, five-session enforcement, protected calibration baselines, user-controlled replacement, progress calculations, archetype comparability, stagnation detection, best-session replay, saved coaching cards, frontend workflows, migrations, and automated verification.
- Out: retraining or retuning visual/audio models, implementing the remaining archetype presets or recommendation quiz, billing, team collaboration, native mobile clients, and unrelated production deployment work.

## Action items
[ ] **Define the durable project and session domain model before changing UI behavior.** Replace the scaffold in `backend/persistence/schema.sql` with versioned migrations for `users`, `projects`, `sessions`, `analysis_results`, `coaching_cards`, and `playbook_entries`. Store project name, goal, default archetype, optional deadline, owner, active baseline session ID, next immutable sequence number, and reset generation. Store each session's project ID, sequence number, archetype key, archetype/scoring-config version, lifecycle state, R2 keys, score snapshot, metric confidence, analysis report, replacement target, and timestamps. Add uniqueness and ownership constraints so sequence numbers cannot be reused or moved between projects.

[ ] **Implement Supabase and R2 adapters behind the existing persistence ports.** Expand `backend/itspeak/persistence.py` into project, session, result, coaching-card, and object-storage interfaces; add local test adapters plus production Supabase/R2 implementations; and move report/video/landmark ownership out of the 24-hour manifest store once a user session is committed. Keep temporary uploads private and expiring, use presigned R2 operations, verify object ownership server-side, and delete R2 objects only through an authenticated project mutation.

[ ] **Add authenticated project-folder APIs and replace frontend mock data.** Add create/list/read/update/reset project endpoints with validation for a required name, optional goal and deadline, enabled archetype, and 300-character context. Add project session-list and detail endpoints ordered by immutable sequence number. Replace `frontend/lib/data.js` fixtures with API-backed loading, empty/loading/error states, and mutations while keeping the current dashboard and project-detail interaction model.

[ ] **Enforce the five-retained-session rule as a server-side transaction.** Before accepting a sixth retained session, return `replacement_required` with eligible Sessions 2-5; require an explicit `replace_session_id`; reject Session 1, cross-project IDs, already-deleted sessions, and stale/concurrent replacement requests. Upload the new recording to temporary storage first, preserve the selected old session if the quality gate or analysis fails, and atomically commit the new session plus deletion of the selected metadata/video/artifacts only after successful analysis. Make the replacement dialog show date, archetype, overall/module scores, and baseline/replay/playbook dependencies before confirmation.

[ ] **Protect Session 1 and implement explicit project reset semantics.** Mark the first successfully committed session in each reset generation as the calibration baseline and prevent ordinary delete/replacement endpoints from removing it. Add a separately authenticated reset action with destructive confirmation that explains it will remove retained sessions, videos, progress history, replay references, and playbook links for that project; then create a new empty generation whose next successful session becomes the new Session 1 baseline. Test aborted uploads so they never consume the baseline slot or sequence number.

[ ] **Create one progress service with explicit comparability rules.** Preserve absolute archetype-calibrated scores in every report, then derive progress as deltas rather than rewriting historical scores. For every available metric and the Vocal/Face/Body aggregates, compute current-vs-previous and current-vs-baseline deltas using only the intersection of non-`insufficient_data` metrics. Mark comparisons `non_normalised` when archetype keys or scoring-config versions differ, never blend those sessions into a single delta, and expose the reason to the frontend. Return a versioned progress payload containing baseline, previous comparable session, deltas, confidence, and comparability metadata.

[ ] **Build the project progress experience.** Add delta cards for previous-session and Session 1 comparisons; an overlaid Vocal/Face/Body radar or trend view; immutable session labels; and visible archetype dividers/annotation icons whenever the archetype changes. Use distinct styling and explanatory copy for `non_normalised`, low-confidence, missing-metric, and baseline-only states. Ensure charts remain readable with five sessions and do not present incomparable polygons as direct improvement.

[ ] **Add configurable stagnation detection and coaching rotation.** Evaluate each metric only across the latest three consecutive completed, comparable sessions with sufficient confidence. Trigger stagnation when none of those sessions improves by the configured minimum meaningful delta; clear the alert after a meaningful improvement, archetype/config change, reset, or insufficient data. Persist which coaching-card fingerprints were shown per session, suppress immediate repeats when another relevant card exists, and pass stagnating metrics to coaching so the model or deterministic fallback proposes an alternative physical technique.

[ ] **Implement Best Session Replay and the Coaching Playbook.** Select the strongest eligible retained session independently for Vocal, Face, and Body using stored aggregate scores, confidence, and the active archetype/config; exclude failed, deleted, insufficient, and non-comparable sessions. Link each winner to authenticated range-enabled R2 playback and its overlays, and recompute winners after replacement/reset. Extend coaching cards with stable IDs, source metric, severity, source session, archetype, and timestamps; let users explicitly save/remove cards; deduplicate playbook entries by normalized card fingerprint; and retain a clear link back to the source session while it exists.

[ ] **Harden concurrency, privacy, and recovery behavior.** Use database transactions or project-level locking for simultaneous uploads, replacement, deletion, and reset; make mutations idempotent; keep Session 1 protection in database/API rules rather than only the UI; enforce Supabase row-level security; prevent cross-user R2 access; reconcile orphaned temporary/R2 objects; and record audit events for replacement, reset, baseline assignment, playbook changes, and archetype switches. Define recovery behavior for R2 deletion failure after a database commit and for database failure after object upload.

[ ] **Verify the complete lifecycle before enabling the feature.** Add backend unit/integration tests for project ownership, first-success baseline assignment, five-session boundaries, concurrent sixth uploads, stale replacement targets, protected baseline deletion, failed replacement analysis, reset generations, comparable/incomparable deltas, missing metrics, stagnation thresholds, replay recomputation, playbook deduplication, RLS, and object cleanup. Add frontend tests for project CRUD, sixth-session replacement, reset confirmation, delta states, archetype dividers, five-session charts, replay seeking, and playbook save/remove. Finish with backend tests, frontend tests/build, migration checks, `git diff --check`, and an end-to-end scenario covering Sessions 1-6 under both same- and changed-archetype paths.

## Open questions

- Is a two-point score increase the initial minimum meaningful improvement for stagnation detection, or should that threshold vary by metric/archetype?
- Should the Coaching Playbook contain only cards the user explicitly saves, as recommended here, or automatically collect every generated card?
